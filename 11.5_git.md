## .gitignore

`.gitignore` is a file in a Git repository that **tells Git which files or folders to ignore**.

- Git will **not track** these files.
- They won’t be added to commits.

some files/directories git should not track

1. **`bin/` & `obj/`** → Never commit compiled binaries; they’re rebuilt automatically.
2. DocFX generated documentation files `_site` , `articles` , `api`

If you accidentally commits a folder before adding `.gitignore`, run:

```
git rm -r --cached [name of file or folder]
git commit -m "Remove build artifacts"
```

## Pull, Push, and Fetch

- **Pull**: Synchronizes your working copy with the remote repository, merging your files with the commits created since the last pull.

- **Fetch**: Doesn’t  modify your **working copy**, updates only your local repository. You have to merge it manually into your work when you’re ready.

 - **Push**: Sends all the commits created locally since the last push to the remote repository

## Troubleshooting some scenarios

**Scenario1**

If you and your teammates are working on the **same branch**, and you’re done for the day, the correct process is:

1. Commit your work locally
2. Pull 
   1. Download changes your teammates made and pushed on remote (repo on GitHub).
   2. There might be conflicts. Resolve conflicts and commit.
3. Push

**Scenario 2**

You have finished working on a feature branch and are ready to push it to the remote repository to create a pull request.

1. Stage and commit your changes on the feature branch.

2. Pull the latest changes from the remote repository to update `master`.

   > This ensures your branch can merge cleanly.

   To do this:

   1. Switch to the `master` branch.
   2. Pull the latest changes from the remote repository.
   3. Switch back to your feature branch.

3. Merge `master` into your feature branch. You want your branch to include the latest changes from `master` before pushing. To do this:

## Rules of thumb

1. `master` stays stable. New work happens on **feature branches**. Changes are merged to master via **pull requests**

2. **Pull frequently**

   **Daily or every time you start working** on your branch. To keep your branch up-to-date with others’ changes so merges don’t become massive or painful.

   go to main branch > pull > go to feature branch > merge main to feature branch > resolve any merge conflicts > commit if needed.

   If more than one person are working on the same branch, you also need to pull from the feature branch.

   pull > resolve any merge conflicts > commit if needed.

3. **Before pushing your changes to create a pull request, always pull**

   go to main branch > pull > go to feature branch > merge main to feature branch > resolve any merge conflicts > commit if needed.

   If more than one person are working on the same branch, you also need to pull from the feature branch.

   pull > resolve any merge conflicts > commit if needed.

## Creating a tag 🏷️ and a release 🚀

Your sprint is finished. The commit ID uniquely defines when your sprint is finished, however,

* Commit IDs are long and unreadable
* Hard to remember
* Not meaningful (you won’t remember what `a7f3c91` represents)

A **tag** in Git:

- Is a label that points to a specific commit
- Lives inside the Git repository
- Is used to mark important versions (e.g., `v1.0`)

Tags do **not** include:

- Descriptions
- Release notes
- Downloadable packaged files

A **release** is a GitHub feature built on top of a tag

When you create a release on GitHub:

- GitHub automatically creates (or uses) a tag
- You can add release notes
- You can attach files (zip, exe, pdf, etc.)
- GitHub generates downloadable source code archives
- It appears nicely in the “Releases” section of your repo
